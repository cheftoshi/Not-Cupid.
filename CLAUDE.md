## Key systems
- **Matching model = ROSTER-FIRST + MULTI-CONNECTION (capacity model, as of 2026-06-07).** The algo NO LONGER auto-assigns; the dashboard shows a **curated roster** and the user **picks**. **BIG CHANGE 6/7: single-match lock RETIRED → users can run up to `MAX_CONNECTIONS = 2` live conversations at once** (`lib/match-actions.ts`). The whole roster stays BROWSABLE even at the cap; to open a NEW chat past the cap the user must **close an existing one** (RosterPicker shows a "close one to open" prompt → routes into `EndMatchDialog`, the reason picker: ghosted / not-vibing). `lib/matching.ts` `rankCandidates()` is the shared eligibility+rank engine (gender → age → radius → ENM cluster → score-threshold-with-wait-decay → equity rank), used by `GET /api/match/roster` + legacy `/api/match` (legacy still uses the old atomic status claim but is NOT in the main flow). **Pick** = `POST /api/match/pick` → now **capacity-gated, NOT an atomic status claim**: caller must have `< 3` live matches, candidate must too, pair not in history/already-live → creates pending match (picker pre-accepted) → nudges candidate. `liveMatchesFor(userId)` + `isMatchLive(m)` (`lib/match-actions.ts`) are the live-count helpers. **`releaseTimedOutMatches()`** lazily expires timed-out pendings on roster/pick/dashboard.
- **⚠️ `users.status` is now INFORMATIONAL ONLY (decoupled from matching, 6/7).** pick + roster gate on the LIVE-MATCH COUNT (capacity), not `status`. Roster pool query no longer filters `status='waiting'` — it surfaces anyone with spare capacity (batch-counts each candidate's live matches, drops those at the cap) and excludes your current live partners + history. `status` is still maintained loosely (matched/waiting) for back-compat with `lib/pools.ts` segmentation + legacy `/api/match`, but status drift can't strand anyone or break picking anymore. The old "INVARIANT: no live match ⇒ status='waiting'" is relaxed (cron/release still set it, harmless if wrong).
- **Dashboard = CONNECTIONS HUB (6/7, `app/dashboard/page.tsx`):** loads ALL live matches → one `MatchCard` each under "your connections", PLUS "the roster" shown alongside ALWAYS (browsable even at cap; picking at cap opens the close-one prompt). MatchReveal fires once for the newest. Dashboard passes `liveConnections`+`maxConnections` to RosterPicker so the close-prompt can list current chats. **Past conversations are READABLE** — history rows link to `/match/[id]` which now renders the chat **read-only** (`readOnly` prop on `ChatRoom`, composer hidden) for ended/expired matches instead of redirecting away. Roster UI = `app/dashboard/roster-picker.tsx`.
- **Roster refinements (5/31):** roster size is 5, but **8 for women seeking men/anyone** (scarce side); women seeking women stay at 5 (thin sought-pool). Roster is **stable, not reshuffled** — persisted via `users.roster_snapshot` + `roster_refreshed_at` with a **12h TTL** (same people show within the window; taken members filtered + backfilled; full recompute past 12h). **auto_rematch is REMOVED** from UI + cron (meaningless when you always get a fresh 5; legacy `/api/rematch-opt-out` + API allowlist entry left, harmless). Profile "active" count requires a genuinely live chat window (not just both_accepted), so it agrees with the dashboard.
