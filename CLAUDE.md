## Key systems
- **Matching model = ROSTER-FIRST (as of 2026-05-31).** The algo NO LONGER auto-assigns one match. Signup + cron just put users in `status='waiting'`; the dashboard shows a **curated roster of ~5** compatible people and the user **picks** one. `lib/matching.ts` `rankCandidates()` is the shared eligibility+rank engine (gender ‚Üí age ‚Üí radius ‚Üí ENM cluster ‚Üí score-threshold-with-wait-decay ‚Üí equity rank, same-intent floated first), used by BOTH `GET /api/match/roster` (top 5) and the legacy `/api/match` (still exists, returns ranked[0], used as a fallback/util ‚Äî NOT called on signup/cron anymore). **Pick** = `POST /api/match/pick` ‚Üí first-pick-wins via atomic conditional `status` claim (claim candidate then self, each only if still 'waiting'; roll back if either fails) ‚Üí creates pending match with picker pre-accepted ‚Üí nudges candidate. `releaseTimedOutMatches()` (`lib/match-actions.ts`) lazily expires the caller's timed-out pending matches on roster/pick so a just-expired user can pick immediately (not wait for cron). Roster UI = `app/dashboard/roster-picker.tsx`; dashboard treats a timed-out/terminal pending match as NOT live so it falls through to the roster.
- **Roster refinements (5/31):** roster size is 5, but **8 for women seeking men/anyone** (scarce side); women seeking women stay at 5 (thin sought-pool). Roster is **stable, not reshuffled** ‚Äî persisted via `users.roster_snapshot` + `roster_refreshed_at` with a **12h TTL** (same people show within the window; taken members filtered + backfilled; full recompute past 12h). **auto_rematch is REMOVED** from UI + cron (meaningless when you always get a fresh 5; legacy `/api/rematch-opt-out` + API allowlist entry left, harmless). Profile "active" count requires a genuinely live chat window (not just both_accepted), so it agrees with the dashboard.
- **Safety (5/31):** **Block & report** ‚Äî `ReportDialog` on match card + chat header ‚Üí `POST /api/report` (reason picker) ends match, blocks the pair from re-matching (match_history), queues for review; admins read/act at `/admin` Reports section (`/api/admin/reports`, block/unblock sets `users.is_blocked`). Matcher + roster exclude is_blocked. **Location privacy** ‚Äî match card/roster never expose exact ZIP/precise distance; show metro label ("Boston, MA") + fuzzy band ("nearby"/"across town"). Date-safety tips in the it's-a-match email.
- **Scoring** (`lib/matching.ts`): **v2 weights (6/5) = values .38 / attachment .27 / traits(HEXACO) .13 / vibes .12 / rapid .10**, renormalized over present components, graceful v1 fallback (old "HEXACO 80% + vibes 20%" retired); adaptive gender-balance threshold; **wait-time threshold decay** (longer wait ‚Üí lower bar, floor 35); **intent prioritization** (prefer same relationship-intent, fall back); **ENM/poly hard cluster** (enm only matches enm, never shown to monogamous & vice versa); **equity rotation** (`equityBonus` in `lib/pools.ts` ‚Äî small under-served bonus). **Per-user match radius** default **15mi**, distance-based (`zipDistanceMiles`), user widens in 15mi steps to 75mi via `/api/profile/expand-radius` (dashboard "widen search" button). Thin-pool nudge email when compatible people exist but all beyond radius.
- **Gender-balance intake gating** (`lib/balance.ts` + `/api/submit` + cron): new over-represented-gender signups in a skewed metro (active >62% one side, pool ‚â•8) get held (`pool_active=false`, `balance_hold_at`); cron releases as the scarce side joins or after a 3-day cap. Held users see the normal positive queue ‚Äî no negative messaging. **KEY INSIGHT from the sim (`scripts/simulate-pool.mjs`): straight-pool stagnation is a SUPPLY problem ‚Äî matches ‚â§ min(M,F); threshold/algo tweaks change WHO matches + quality, never the COUNT. Only adding women moves it. So the real fix is women-first recruiting/pricing (growth, not code).**
- **Match accept** (`lib/match-actions.ts` `acceptMatch`): email link AND in-app button both go through ONE shared activation ‚Äî sets accepted flag, first-accept nudges the other ("interested, accept to connect"), mutual accept ‚Üí status=both_accepted + opens chat + sends it's-a-match contact emails. **Expiry timings:** unaccepted pending match expires **24h** after creation (`expires_at`, set on create); accepted **chat** is inactivity-based at **36h** (`CHAT_INACTIVITY_MS`), every message slides it forward (`/api/messages`); cron closes silent chats + sweeps timed-out pending matches.
- **Pools/segmentation** (`lib/pools.ts`): computed-live 2D segments ‚Äî intent (serious/casual/enm/open from `relationship_style` + `future` vibe) √ó tier (A/B/next from session recency). Penalty states: cooldown / banned. Admin view at `/admin`.
